# مزامنة عمل Claude وCodex — OZK TOBACCO

هذا الملف هو المرجع المشترك لأي مساعد ذكاء اصطناعي يعمل على المشروع. عند اختلاف ذاكرة جلسة قديمة مع هذا الملف أو مع الكود الحالي، تكون الأولوية للكود الحالي ثم لهذا الملف ثم لذاكرة الجلسة.

## قبل بدء أي عمل

1. اقرأ `AGENTS.md` و`CLAUDE.md` وهذا الملف.
2. اقرأ `AI_ACTIVE_TASK.json` و`AI_HANDOFF.md` لمعرفة مَن يعمل وأين انتهى.
3. تأكد أن مجلد العمل هو مستودع `tobacco-web` الحقيقي.
4. افحص حالة Git. لا تمسح تغييرات موجودة ولا تخلطها مع مهمة أخرى.
5. إذا كان المجلد نظيفاً، نفّذ `git pull --rebase origin main` قبل التعديل.
6. راجع آخر التعديلات ذات الصلة في `git log` ولا تعتمد على محادثة قديمة وحدها.
7. احجز المهمة على `main` قبل إنشاء فرع العمل: `powershell -File tools/ai-work-coordination.ps1 -Action Claim -Owner Codex -Task "وصف المهمة" -Files src/app.js`.
8. انشر الحجز فوراً بـcommit وpush عاديين على `main` كي تراه الجلسات والأجهزة الأخرى، ثم أنشئ فرعاً منفصلاً للمهمة.

إذا كان القفل `active` فلا تبدأ عملاً متداخلاً. يجوز العمل بالتوازي فقط عندما تكون الملفات والنطاقات منفصلة بوضوح، وبعد تسجيل ذلك في دفتر التسليم.

## أثناء العمل

- العربية هي اللغة الرسمية للواجهة والتواصل، مع RTL ونص واضح على iPhone.
- Windows هو بيئة التطوير والمزامنة، وiPhone هو جهاز الاستخدام الأساسي.
- حافظ على بنية HTML/JavaScript/CSS الحالية؛ لا تدخل React أو Vite أو macOS من دون طلب صريح.
- الأمين هو المصدر المحاسبي الموثوق. لا تصحح رقماً يدوياً قبل فحص المصدر والمزامنة.
- لا تقرأ أو تنسخ `.env` أو مفاتيح أو كلمات مرور أو connection strings.
- لا تنشر تقارير أو أرصدة أو بيانات زبائن على استضافة عامة.
- لا تزامن فواتير المشتريات مع الأمين؛ المسموح تصدير يدوي يبدأه المستخدم فقط.
- إذا اكتشفت قاعدة محاسبية أو سبب عطل جديد، وثّقه في قسم سجل القرارات أدناه باختصار مع التاريخ والملفات المتأثرة.
- استخدم فرعاً منفصلاً لكل إصلاح أو ميزة. لا يعمل مساعدان على الفرع نفسه.

## قبل إعلان النجاح

1. شغّل `npm.cmd run check` على Windows.
2. اختبر الحسابات بمقارنة مستقلة مع مصدر الأمين عند تعديل المخزون أو الأرصدة أو الفواتير.
3. عند تعديل HTML أو CSS أو JavaScript منشور، ارفع `CACHE_NAME` في `public/service-worker.js`.
4. اعرض `git diff --check` وراجع أن التغيير يخص المهمة فقط.
5. لا تدّع نجاح النشر أو iPhone من دون دليل فعلي.
6. لا تعمل commit أو push أو PR إلا بطلب صريح من المستخدم.
7. بعد دمج فرع المهمة، ارجع إلى `main` واسحب آخر نسخة، ثم أغلق القفل وحدّث التسليم: `powershell -File tools/ai-work-coordination.ps1 -Action Complete -Owner Codex -Task "وصف المهمة" -Note "النتيجة والفحوص"`.
8. انشر تحديث التسليم وإخلاء القفل فوراً بـcommit وpush عاديين على `main`.

قفل المهمة وتسليمها هما الاستثناء التنظيمي الوحيد الذي يُنشر مباشرة على `main`؛ كود الموقع والإصلاحات والميزات تبقى دائماً على فروع مستقلة وتُدمج بعد الفحص.

## قواعد محاسبية لا يجوز كسرها

- المخزون يُحسب من حركات الفواتير وأعلام `bIsInput` و`bIsOutput`، لا من `ms000` بعد تدوير السنة.
- كشف الحساب يستعمل `balance` بترتيب كشف الأمين، بينما مستند الفاتورة يستعمل `balanceChrono` و`docPrev`/`docNew` للسند الكامل.
- أساس سعر سطر الفاتورة قد يكون كرتونة أو كروز ويُحسم عبر `invoicePriceBasis` بمطابقة الإجمالي.
- مرتجع المبيعات هو `BillType=3` وقد يحتاج مطابقة بالتاريخ والمبلغ عند غياب `BiGUID`.
- عند رصيد قديم، افحص توقيت مهمة المزامنة وقارن `customerAccountGuid` مع `ac000` قبل تغيير المنطق.
- نشرة الجملة بالدولار تعرض الصنف فقط إذا كان مخزونه يساوي وحدة ثانية كاملة واحدة على الأقل (`stock_qty / unit2_factor >= 1`). نشرة المفرق بالسوري تقبل أي مخزون موجب، ويجوز تسعيرها بلا سعر دولار.
- سعر المفرق (`price_payload.retail.price`) مستقل تماماً عن سعر الجملة: يجوز حفظه عندما يكون `unit2_price = 0`، وتستعمل نشرة السوري هذا السعر، بينما يبقى الصنف خارج نشرة الدولار حتى يُسعّر للجملة وتتوافر وحدة ثانية كاملة.
- قائمة «أسعار النشرة» داخل الموقع تطابق النشرة العامة: تستبعد الوزاري، تعرض الدمج نفسه للمعسل والأصناف المتشابهة، وتطبّق شرط الوحدة الثانية الكاملة في وضع الجملة فقط. حفظ سعر سطر مدمج يحدّث جميع أصنافه الأصلية.
- مولّد النشرات يجب أن يدفع ملفات الأسعار من دون `[skip ci]` كي يشغّل `pages.yml` وينشر الأسعار الجديدة فعلياً؛ مسار توليد النشرة محصور بتغيير `scripts/price-data.json` فلا يسبب حلقة تشغيل عند دفع ملفات `public/downloads`.
- مزامنة مخزون النشرة تقارن العدد الصحيح للوحدة الثانية الكاملة (`floor(stock_qty / unit2_factor)`) بين الأمين والمخزون المنشور، وتحدّث عند تغيّره أو عند الانتقال بين مخزون موجب وصفر؛ لذلك ينطبق الحد تلقائياً على الكرتونة والطرد والشرحة حسب معامل كل صنف.
- تقرير المخزون يعرض كل صنف ذي كمية موجبة مستقلاً بلا دمج (بما فيها أصناف المعسل)، ويعرض النافد فقط إذا كان له مبيع فعلي حديث. تصنيف «قريب من النفاد» يعتمد تغطية آخر 30 يوماً من حركة المبيع، مع إبقاء حدود الأمان الخاصة للماستر والغلواز؛ الصنف البطيء بلا مبيع حديث لا يُصنّف قريب النفاد لمجرد أن كميته دون حد ثابت.
